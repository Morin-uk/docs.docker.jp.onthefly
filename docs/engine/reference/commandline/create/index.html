docker create

説明

コンテナーを新規生成します。

利用方法

docker create [オプション] IMAGE [COMMAND] [ARG...]

追加説明

docker create コマンドは、指定されたイメージの上に書き込み可能なコンテナーを生成します。 そして指定されたコマンドが実行できるように準備します。 この後にコンテナー ID が STDOUT に出力されます。 これは docker run -d に似ていますが、ただしこのコマンドではコンテナーは起動しません。 これに続けて docker start <コンテナーID> コマンドを実行すれば、どのような時点でもコンテナーを起動させることができます。

事前にコンテナー設定を行っておいて、必要となるときまでコンテナーを起動させないでおきたい場合に、このコマンドが役立ちます。 新たに生成されたコンテナーの初期状態は created（生成済み）です。

詳細は run コマンドや Docker run リファレンス を参照してください。

本コマンドの利用例については、以下に示す 利用例の節 を参照してください。

利用例

コンテナーの生成と起動

$ docker create -t -i fedora bash

6d8af538ec541dd581ebc2a24153a28329acb5268abe5ef868c1f1a261221752

$ docker start -a -i 6d8af538ec5

bash-4.2#

ボリュームの初期化

v1.4.0 現在、コンテナーボリュームはdocker create処理フェーズにおいて初期化されます。 （docker runも同じです。） つまりcreateによってdataボリュームコンテナーが生成されるので、これを別のコンテナーが利用できることを意味します。

$ docker create -v /data --name data ubuntu

240633dfbb98128fa77473d3d9018f6123b99c454b3251427ae190a7d951ad57

$ docker run --rm --volumes-from data ubuntu ls -la /data

total 8
drwxr-xr-x  2 root root 4096 Dec  5 04:10 .
drwxr-xr-x 48 root root 4096 Dec  5 04:11 ..

同様にしてcreateは、ホストディレクトリをバインドマウントするコンテナーを生成します。 そしてこれに続くコンテナーがこれを利用できるようにします。

$ docker create -v /home/docker:/docker --name docker ubuntu

9aa88c08f319cd1e4515c3c46b0de7cc9aa75e878357b1e96f91e2c773029f03

$ docker run --rm --volumes-from docker ubuntu ls -la /docker

total 20
drwxr-sr-x  5 1000 staff  180 Dec  5 04:00 .
drwxr-xr-x 48 root root  4096 Dec  5 04:13 ..
-rw-rw-r--  1 1000 staff 3833 Dec  5 04:01 .ash_history
-rw-r--r--  1 1000 staff  446 Nov 28 11:51 .ashrc
-rw-r--r--  1 1000 staff   25 Dec  5 04:00 .gitconfig
drwxr-sr-x  3 1000 staff   60 Dec  1 03:28 .local
-rw-r--r--  1 1000 staff  920 Nov 28 11:51 .profile
drwx--S---  2 1000 staff  460 Dec  5 00:51 .ssh
drwxr-xr-x 32 1000 staff 1140 Dec  5 04:01 docker

コンテナーごとにストレージドライバーのオプションを設定します。

$ docker create -it --storage-opt size=120G fedora /bin/bash

上の例において（そのサイズ設定）は、コンテナー生成時に rootfs のサイズを 120 G に設定しています。 そのオプションはdevicemapper、btrfs、overlay2、windowsfilter、zfsの各グラフドライバーにおいてのみ利用可能です。 そして devicemapper、btrfs、windowsfilter、zfsの各ドライバーにおいて、設定可能なサイズはデフォルトの BaseFS サイズ以下でなければなりません。 ストレージドライバーoverlay2では、バックエンドのファイルシステムがxfsであって、マウントオプションpquotaを使ってマウントされている場合に限って、サイズオプションの指定が可能です。 こういった条件のもと、バックエンドのファイルシステムサイズ以内であれば、どのようなサイズ設定でも可能です。

コンテナーに対する分離技術方式の指定 (--isolation)

このオプションは Docker コンテナーを Windows 上で実行する場合に有用なものです。 --isolation=<設定値>は、コンテナーの分離技術（isolation technology）方式を指定します。 Linux 上においては、Linux 名前空間を利用するdefault指定のみがサポートされます。 Microsoft Windows の場合、以下の値を設定することができます。

設定値 内容説明
default Docker デーモンの--exec-optの指定に従います。daemonに分離技術が指定されていない場合、Microsoft Windows において、デーモンが Windows サーバー上の稼動であればprocessを、あるいは Windows クライアント上であればhypervをそれぞれデフォルトで採用します。
process 名前空間による分離のみを行います。
hyperv ハイパーバイザー Hyper-V によるパーティションベースの分離を行います。

--isolationフラグに値を設定しなかった場合、--isolation="default"と指定したことになります。

動的生成デバイスの扱い (--device-cgroup-rule)

コンテナーが利用可能なデバイスは、コンテナーの生成時に割り当てられます。 割り当てられたデバイスは cgroup.allow ファイルに加えられ、またコンテナーが起動したときにコンテナー内に生成されます。 ここで問題になるのが、実行中コンテナーに対して新たなデバイスを追加したい場合にはどうするかということです。

解決方法の 1 つとして、コンテナーにもっとゆるいルールを設定して、幅広くデバイスにアクセスできるようにする方法があります。 たとえば、キャラクターデバイスとしてメジャー番号42、マイナー番号は何でも取るような（新たなデバイスとして出てきたら加えられる）デバイスにアクセスする必要があるとします。 この場合には、以下のようなルールを適用することになるはずです。

docker create --device-cgroup-rule='c 42:* rmw' -name my-container my-image

そして必要なデバイスが追加された際にdocker exec my-container mknod newDevX c 42 <minor>を実行するようなスクリプトを、udevから実行するように設定します。

メモ: はじめから存在しているデバイスも、create/run コマンドにおいて明示的に追加することが必要です。
